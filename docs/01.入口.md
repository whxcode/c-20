# 01.入口

这份文档负责说明整套机制从哪里进入，以及 `MkModelBase`、`MkCommitService`、`MkDocument`、`MkHistoryManager` 之间如何串起来。后续文档会继续展开属性收集、多用户协作和最小示范代码。

## 总览

核心设计可以理解为两条链路：

```text
本地修改链路：
UI/业务代码
  -> MkComponent::invokeProperty / MkModelBase::setValue
  -> 具体 setXxx 宏保存 old/new
  -> MkModelBase::triggerChanged
  -> MkCollectorImpl 收集 old/new
  -> MkCommitService::commit
  -> MkCollectorImpl::commit 生成 {undo, redo}
  -> MkHistoryManager 保存历史并把 redo 发给 IO

回放/同步链路：
History undo/redo 或远端 patch
  -> MkCommitService::mergePatches
  -> MkMergePatch 解码 patch
  -> props()->setValue 写回 Model
  -> DocumentMergeDelegate 更新组件树
  -> triggerUpdated 计算布局/副作用
```

最关键的分工：

- `MkModelBase`：属性变化入口，负责把 `oldValue/newValue` 交给收集器，并通知绑定组件。
- `MkDataProvider/MkDataSource`：持有所有 `MkModelBase`，把同一个 `MkCollectorImpl` 注入每个 model。
- `MkCollectorImpl`：暂存本轮变更，最终生成一对 patch：`undo` 与 `redo`。
- `MkCommitService`：提交时机控制、事务、触发文档副作用、网络发送、undo/redo 执行入口。
- `MkHistoryManager`：只管理历史栈，不直接懂文档；真正执行 patch 通过回调交给 `MkCommitService`。
- `MkDocument`：文档对象与组件树总入口，提供 `getDataProvider()`、`triggerUpdated()`、`beginUpdate()/endUpdate()`。

## 属性修改入口

属性面板改属性时，入口大致是：

```text
MkPropPanelProvider::setPropPanelElement
  -> MkComponent::invokeProperty
  -> MkLayerBase::onInvokeProperty
  -> MkComponent::onInvokeProperty
  -> MkModelBase::setValue
  -> MkPropValue::SetValue
  -> 具体 model 的 setXxx
  -> MkModelBase::triggerChanged
```

代码上的统一入口是：

- `mk-provider/src/models/MkModelBase.cpp:40`：`MkModelBase::setValue(key, value)`。
- `mk-provider/src/models/MkModelBase.cpp:82`：`MkModelBase::triggerChanged(scope, key, oldValue, newValue)`。
- `mk-provider/include/models/MkModelDefined.h:170`：宏生成的 `setXxx` 会比较新旧值、保存旧值、写入新值，然后调用 `triggerChanged`。

一个典型 setter 逻辑等价于：

```cpp
void setName(const std::string& value) {
    if (compareValue(getName(), value)) {
        return;
    }

    auto oldValue = getName();
    fProperties.setValue(PropKey::kName, value);
    triggerChanged(MkModelScope::kRepaint, PropKey::kName, &oldValue, &getName());
}
```

因此，实现这类机制时，必须保证所有属性修改都走统一 setter。不要绕过 setter 直接改字段，否则无法自动记录 undo/redo。

## 三个核心对象的关系

### MkDocument

`MkDocument` 是“文档数据 + 组件树 + 文档副作用”的总门面：

- `mk-document/include/document/MkDocument.h:77`：通过 `getDataProvider()` 暴露底层数据。
- `mk-document/include/document/MkDocument.h:142`：通过 `triggerUpdated()` 触发布局、symbol、资源版本等副作用。
- `mk-document/src/document/MkDocumentImpl.cpp:198`：构造时创建 `MkDataProvider`。
- `mk-document/src/document/MkDocumentImpl.cpp:222`：创建 `OperationCollector` 并挂到组件树，负责组件 add/remove/modify 的文档级副作用。

### MkCommitService

`MkCommitService` 是 editor 层的提交控制器：

- `mk-editor/src/service/MkCommitService.cpp:19`：构造时创建/绑定 `MkHistoryManager`。
- `mk-editor/src/service/MkCommitService.cpp:33`：把 `MkHistoryManager` 的执行回调绑定到 `MkCommitService::onPatchExecutor`。
- `mk-editor/src/service/MkCommitService.cpp:77`：`commit()` 是普通提交入口。
- `mk-editor/src/service/MkCommitService.cpp:214`：`doCommit()` 先 `triggerUpdated()`，再从 collector 取 patch，并交给 history。
- `mk-editor/src/service/MkCommitService.cpp:294`：`undo()` 委托给 `MkHistoryManager::undo()`。
- `mk-editor/src/service/MkCommitService.cpp:299`：`redo()` 委托给 `MkHistoryManager::redo()`。

注意：`MkCommitService` 不直接保存每个属性的 old/new；它只在提交时向 `document->getDataProvider()->getCollector()` 要本轮 patch。

### MkHistoryManager

`MkHistoryManager` 是纯历史栈管理器：

- `mk-editor/src/service/MkHistoryManager.cpp:128`：`commit(pair, options)` 把 `{undo, redo}` 放进历史或事务暂存。
- `mk-editor/src/service/MkHistoryManager.cpp:141`：`undo()` 取当前历史项的 undo buffer，通过 `fOnExecutor` 回调执行。
- `mk-editor/src/service/MkHistoryManager.cpp:164`：`redo()` 取 redo buffer，通过同一个 executor 执行。
- `mk-editor/src/service/MkHistoryManager.cpp:105`：事务开始；多个 commit 可先进入 stash。
- `mk-editor/src/service/MkHistoryManager.cpp:114`：事务结束；stash 合并成一条历史。

三者关系可以简化为：

```text
MkCommitService
  owns MkHistoryManager
  uses MkDocument

MkDocument
  owns MkDataProvider
    owns MkCollectorImpl

MkHistoryManager
  stores patch pairs
  calls MkCommitService executor to apply undo/redo
```
