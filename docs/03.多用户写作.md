# 03.多用户写作

这份文档说明“用户 B 修改图层，用户 A 如何收到变化，并且不会形成循环收集”。文件名沿用当前命名；这里的“写作”实际描述的是多用户协作/同步。

## 同进程内的多个观察者

如果“用户 A/B”只是代码里的两个观察者或 UI 控件，它们共享同一个 `MkModelBase`：

```text
B 调用 model->setValue(...)
  -> setter 改 model
  -> triggerChanged
  -> MkComponent::modelChangedCallback
  -> MkComponent::onDataChanged
  -> notifyObservers(kDataChanged, key)
  -> A 的视图/缓存收到通知
```

这种场景不会天然循环，前提是 observer 收到通知后只刷新视图，不再次无条件 `setValue`。若 observer 需要反写，必须先比较值，或者使用 `runWithoutCollect/closeCollector` 这类保护。

## 多端协同的 Patch 流程

B 本地 commit 后：

```text
B collector->commit()
  -> 得到 patchPair
  -> MkHistoryManager::commit()
  -> fOnPostBuffers({redo})
  -> MkCommitService::onPostBuffersToIO()
  -> pipeline send
```

A 收到 patch 后：

```text
pipeline onReceive
  -> fReceiveStash.receiveBuffers()
  -> MkCommitService::onExecIOPatches()
  -> mergePatches(kIO, buffers)
  -> MkMergePatch::merge(..., isCloseCollector = true)
  -> collector.closeCollector()
  -> 写入 A 的 model
  -> collector.openCollector()
  -> stepCommit()
```

## 防循环机制

关键防循环点：

- 网络 IO patch 合并默认 `isCloseCollector = true`。
- `MkMergePatch::merge()` 在 `isCloseCollector` 为 true 时调用 `collector->closeCollector()`。
- collector 关闭期间，`MkModelBase::triggerChanged()` 虽然会通知组件，但 `fCollector->canCollector()` 为 false，不会再次记录成新的本地提交。
- 合并结束后再 `openCollector()`，然后 `stepCommit()` 只触发刷新和事件，不把远端 patch 当成本地新历史提交。

对应代码：

- `mk-editor/src/service/MkCommitService.cpp:330`：收到 IO patch 后 `onExecIOPatches()`。
- `mk-editor/src/service/MkCommitService.cpp:331`：`mergePatches(MkPatchExecuteMode::kIO, buffers)`，默认关闭 collector。
- `mk-provider/src/state/MkMergePatch.cpp:38`：`collector->closeCollector()`。
- `mk-provider/src/state/MkMergePatch.cpp:45`：合并后 `collector->openCollector()`。

这就是“不循环收集”的主要机制：远端同步只更新本地数据和视图，不生成新的本地 redo patch 再发回服务器。

## 与 undo/redo 的差异

远端 patch 和本地 undo/redo 都会写回 model，但 collector 策略不同：

- 远端 IO patch：通常关闭 collector，避免回声循环。
- 本地 undo/redo：可以不关闭 collector，让回放产生的文档副作用继续被收集。
- 普通本地修改：正常打开 collector，等待 commit 生成 undo/redo。

这三个场景必须分开处理，否则很容易出现“远端 patch 被当成本地操作再次广播”的问题。
